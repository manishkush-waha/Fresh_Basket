Introduction

Fresh Basket is a comprehensive web application designed to revolutionize the way users shop for groceries online. With a focus on convenience, efficiency, and user satisfaction, this platform aims to bridge the gap between traditional grocery shopping and modern digital solutions.

Purpose of the Report

The purpose of this report is to provide a detailed overview of the Fresh Basket project, including its objectives, architecture, development process, and future scope. It serves as a comprehensive guide for stakeholders, developers, and users to understand the various aspects of the application.

Scope of the Web Application

Fresh Basket is designed to cater to a wide range of users, from busy professionals to homemakers, by offering a seamless online grocery shopping experience. The application includes features such as product browsing, personalized recommendations, secure checkout, and real-time order tracking.

Target Audience

The primary target audience for Fresh Basket includes individuals and families who prefer the convenience of online shopping. Additionally, it aims to support local grocery stores by providing them with a digital platform to reach a broader customer base.

Database Design

Database Management System

Fresh Basket uses MongoDB as its database management system. MongoDB is a NoSQL database that provides flexibility and scalability, making it an ideal choice for modern web applications. It stores data in a JSON-like format called BSON, which allows for dynamic schemas and hierarchical data structures.

ER Diagram

The Entity-Relationship (ER) diagram for Fresh Basket represents the relationships between various entities in the database. Key entities include Users, Products, Orders, and Categories. Relationships such as "Users place Orders" and "Orders contain Products" are depicted to illustrate how data is interconnected.

The ER diagram helps in visualizing the database structure and serves as a blueprint for implementation.

Database Schema

Tables and Fields

The database schema for Fresh Basket is designed to efficiently store and retrieve data. Below are the key collections and their fields:

  • Users:
    • _id: Unique identifier for the user (ObjectId).
    • name: Full name of the user (String).
    • email: Email address of the user (String, unique).
    • password: Hashed password for authentication (String).
    • address: Delivery address of the user (String).
    • role: Role of the user (e.g., customer, admin) (String).
    • createdAt: Timestamp of account creation (Date).
  • Products:
    • _id: Unique identifier for the product (ObjectId).
    • name: Name of the product (String).
    • description: Detailed description of the product (String).
    • price: Price of the product (Number).
    • category: Category to which the product belongs (ObjectId, references Categories).
    • stock: Quantity available in stock (Number).
    • image: URL of the product image (String).
    • createdAt: Timestamp of product addition (Date).
  • Orders:
    • _id: Unique identifier for the order (ObjectId).
    • user: User who placed the order (ObjectId, references Users).
    • products: List of products in the order (Array of Objects with productId and quantity).
    • totalPrice: Total price of the order (Number).
    • status: Current status of the order (e.g., pending, shipped, delivered) (String).
    • createdAt: Timestamp of order placement (Date).
  • Categories:
    • _id: Unique identifier for the category (ObjectId).
    • name: Name of the category (String).
    • description: Description of the category (String).

Relationships

The relationships between collections are as follows:

  • Users and Orders: A user can place multiple orders, establishing a one-to-many relationship.
  • Orders and Products: An order can contain multiple products, creating a many-to-many relationship. This is implemented using an embedded array of product objects within the Orders collection.
  • Products and Categories: Each product belongs to a single category, forming a many-to-one relationship.

Data Validation and Constraints

Data validation and constraints are implemented using Mongoose schemas to ensure data integrity. Key validations include:

  • Unique Fields: Fields such as email in the Users collection are marked as unique to prevent duplicates.
  • Required Fields: Essential fields like name, email, and password in the Users collection are marked as required.
  • Data Types: Fields are strictly typed (e.g., price as Number, createdAt as Date) to ensure consistency.
  • Custom Validations: Custom validation functions are used for fields like email to ensure proper formatting.
  • Default Values: Fields like createdAt are assigned default values using Mongoose schema options.

By leveraging Mongoose's powerful schema validation features, Fresh Basket ensures that only valid and consistent data is stored in the database.

Authentication and Authorization

Fresh Basket implements robust authentication and authorization mechanisms to ensure secure access to the application. Key aspects include:

Session vs Token-Based

The application uses token-based authentication with JSON Web Tokens (JWT) for its scalability and stateless nature. Key advantages include:

  • Session-Based: Suitable for small-scale applications but requires server-side storage and management.
  • Token-Based: Ideal for modern web applications, as tokens are stored on the client side and sent with each request.

JWTs are signed and encoded, ensuring data integrity and security. They are stored in HTTP-only cookies to prevent XSS attacks.

Role Management

Role-based access control (RBAC) is implemented to restrict access to specific resources based on user roles. Key roles include:

  • Admin: Full access to manage users, products, and orders.
  • Customer: Access to browse products, place orders, and view order history.

Middleware functions are used to verify user roles before granting access to protected routes.

Security Considerations

Common Vulnerabilities Handled

XSS, CSRF

To mitigate Cross-Site Scripting (XSS) attacks, Fresh Basket ensures that all user inputs are sanitized and escaped before rendering them on the frontend. Additionally, Content Security Policy (CSP) headers are implemented to restrict the execution of unauthorized scripts.

For Cross-Site Request Forgery (CSRF) protection, the application uses CSRF tokens. These tokens are generated on the server and validated with each request to ensure that only legitimate requests are processed.

SQL Injection

Although Fresh Basket uses MongoDB, which is not directly vulnerable to traditional SQL injection, it still implements robust query validation to prevent NoSQL injection attacks. Mongoose ORM is used to validate and sanitize inputs before executing database queries, ensuring that malicious payloads are not processed.

HTTPS and Secure Headers

Fresh Basket enforces HTTPS for all communications between the client and server to ensure data is transmitted securely. Secure headers are configured using middleware such as Helmet.js to add the following protections:

  • Strict-Transport-Security (HSTS): Ensures that browsers only communicate with the server over HTTPS.
  • X-Content-Type-Options: Prevents MIME type sniffing.
  • X-Frame-Options: Protects against clickjacking attacks by disallowing the application to be embedded in iframes.
  • Content-Security-Policy (CSP): Restricts the sources from which scripts, styles, and other resources can be loaded.

User Data Protection

User data is protected through encryption and secure storage practices. Key measures include:

  • Password Hashing: User passwords are hashed using bcrypt before being stored in the database.
  • Data Encryption: Sensitive data, such as payment information, is encrypted using AES-256 encryption.
  • Access Control: Role-based access control (RBAC) ensures that only authorized users can access sensitive data.
  • Regular Audits: Periodic security audits are conducted to identify and address vulnerabilities.

Rate Limiting and Throttling

To prevent abuse and ensure fair usage of resources, Fresh Basket implements rate limiting and throttling mechanisms. These measures include:

  • Rate Limiting: Limits the number of requests a user can make within a specific time frame using libraries like express-rate-limit.
  • IP Blocking: Temporarily blocks IP addresses that exceed the allowed request limit.
  • API Throttling: Ensures that API endpoints are not overwhelmed by excessive requests, maintaining application performance.
  • Monitoring: Tools like New Relic or Datadog are used to monitor traffic patterns and detect anomalies.

These security measures collectively ensure that Fresh Basket remains a secure and reliable platform for its users.
